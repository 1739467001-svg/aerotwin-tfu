# 开发历程 · 天府 TWIN 机场数字孪生

> 一份按时间线整理的「人 × AI 结对共建」开发日志。
> 记录本项目如何从一个**纯逻辑仿真内核**，一步步长成「会发光、有昼夜、会下雨、能多端触控操作」的网页版**成都天府国际机场（TFU）数字孪生**。
>
> **协作模式**：我（项目主理人）提出诉求与方向 → Claude（AI 结对工程师）给带「性价比 / 风险」标注的方案、写代码、跑单元测试、`git commit` + `push`。
> 每个特性**独立一次提交**，可逐步回溯；全程 **9/9 单元测试**守护仿真内核不回归。

## 一、技术底座与三条铁律

| 铁律 | 含义 |
|---|---|
| **运行时零依赖、零构建** | 双击 `index.html` 即运行；唯一外部依赖是 CDN 上的 three.js。后期效果器 `fx.js` 全程只用核心 three，不引入任何额外 CDN。 |
| **仿真 / 渲染解耦** | `sim.js` 是纯逻辑、零依赖、浏览器与 Node 通用的内核；`index.html` 只把仿真状态投影成 3D 画面。视觉怎么改都不动内核。 |
| **可复现 + 可回归** | 固定随机种子（`20260611`）+ 固定步长推进，任意机器/帧率结果一致；`npm test`（Node 内置测试器，零 npm 依赖）覆盖确定性、活性、资源互斥、A/B 结论与 KPI 基准。 |

## 二、协作方法论（「如何做出来的」关键）

这套流程是项目能快速成形又不失控的核心：

1. **方向先行，菜单选择**：每进入一个大阶段，我先抛出诉求；Claude 不直接闷头写，而是给一份**带性价比与风险标注的方案菜单**，我来拍板（很多时候是「都要做」），再逐项落地。
2. **一次一提交**：每个特性单独 `commit` + `push`，提交信息记录「做了什么 + 权衡」，方便回溯与展示。
3. **AI 的诚实边界**：云端开发环境**装不了浏览器、CDN 被墙**，Claude **无法自己截图验证视觉效果**。对此采取的工程策略是——
   - 所有可视参数集中、可在浏览器控制台**实时热调**（`AEROTWIN_FX.cfg.* + AEROTWIN_FX.apply()`）；
   - 着色器/后期异常**优雅回退**为直接渲染，**绝不黑屏**；
   - **高风险项默认关闭**（如 SSAO），交由我开启评估；
   - 由我用眼睛验收、反馈「症状词」（太亮/太糊/穿模/坡度反了…），Claude 据此精调。
4. **内核稳定优先**：所有视觉与 UI 改动**全程不碰 `sim.js`**（除非显式改内核并同步更新 golden 基准），保证 A/B 仿真结论始终可信、28 次提交里测试一直是 9/9。

## 四、分阶段详述

### 第 0 阶段 · 奠基（`aa10c95` → `8a1717c`）

先有「能跑的仿真」，再谈好看。这一阶段确立了项目的骨架与工程纪律：

- **v1.0**：以成都天府机场为参考，单份固定时刻表、两个独立仿真实例并行推进，对比 **FCFS 基线** vs **智能调度** 两种跑道/机位策略；雷暴关闭跑道事件；白天/夜间主题。
- **内核抽离**：把仿真逻辑从页面里抽成 `sim.js`（纯函数、零依赖、Node 可测），补上单元测试；并修掉一个隐蔽 bug——用 `!occupiedBy` 判空会把 **0 号航班**占用的资源误判为空闲（falsy-0），导致跑道/机位双占，改为一律 `=== null`。
- **滑行道防撞**：同车道、同向、前方近距即停的跟车间隔，防止飞机在滑行道上叠合。

> 🧱 这一阶段的产物——「解耦的内核 + 测试 + 零依赖」——是后面能放心大改视觉的底气。

### 第 1 阶段 · 视觉觉醒：光影四部曲（`e8d8af3` → `c65c21d`）

> 我的原话：**「关于模拟视觉方面，有没有什么建议？因为我认为模拟的视觉效果是我首要做这个项目的原因。」**

Claude 给了一份按性价比分档（S/A/B）的建议，并提出四个方向让我选；我答 **「都要做，一步一步来，做好一个再做下一个」**。于是一次一个、各自成提交：

1. **光影基础包**（`e8d8af3`）——新建 `fx.js` 自研后期合成器：Bloom 泛光 + 移轴景深（缩微模型质感）+ FXAA；并启用 PCFSoft 太阳软阴影。**关键决策**：为守住「零依赖」承诺，不用 three 的 examples 后期，全部自己用核心 three 的 RenderTarget + 全屏 ShaderPass 实现。
2. **昼夜循环 + 真实天空**（`f8410ed`）——太阳随虚拟时钟连续划过天空，大气散射式渐变（晨光/正午/黄金时刻/日落/星夜），平行光与可见太阳同步真实日位，**阴影方向随太阳走**；月亮与星空按入夜程度渐显；按钮改为「自动/固定白天/固定夜间」三态。
3. **飞机精细化 + 航行灯**（`45b22d2`）——翼下双发、驾驶舱风挡、客舱舷窗；红/绿翼尖航行灯、白色频闪、防撞信标、起降灯、起飞尾流。改为**共享几何/材质**，~40 架零额外开销。
4. **雷暴天气可视化**（`c65c21d`）——把原本只「关闭跑道」的雷暴事件接上视觉：阴云压顶、雨幕、闪电、湿地面，联动既有按钮。

### 第 2 阶段 · 电影感进阶（`45006bb` → `c4968ce`）

延续「都要做」，我把上一轮 Claude 提的延伸方向全点了：

- **相机预设 + 跟随运镜**（`45006bb`）：总览/塔台俯瞰/02R 进近端/正俯视，平滑过渡；「🎯 跟随」选中飞机运镜。
- **世界空间凝结尾迹**（`9433d37`）：起飞爬升沿真实航迹播撒、脱离飞机留在空中、渐隐扩散的气团（按距离步进，与倍速解耦）。
- **玻璃幕墙环境反射**（`e40f9b6`）：自写菲涅尔天空反射着色器，复用天空颜色对象，随昼夜/雷暴实时变化，零渲染开销。
- **ACES 色调**（`c4968ce`）：高光柔化 + 轻度 sRGB，并刻意保留深空暗部配色。

### 第 3 阶段 · 填满机场（`a2eb91b` → `5321607`）

> 我的原话：**「你觉得机场、人、接驳车，还有哪些元素可以增加？我想优先提升模拟视觉。」**

Claude 指出最「空」的是近机位停机坪，给了四组场景元素菜单；我再次 **「都要做，一个一个慢慢来」**：

- **停机坪生活**（`a2eb91b`）：登机廊桥随 `atGate` 平滑对接、推出时收回；地服车辆（行李拖车小火车/客梯餐车/GPU）+ 地勤 + 轮挡；黄色机位引导线 + 地面机位号。
- **机坪 & 进近灯光**（`4373596`）：高杆泛光灯、PAPI 坡度灯、跑道阈值斑马线/跑道号/接地带标线、进近顺序闪灯（兔子灯）、随风/雷暴摆动的风向袋。
- **天际线与远景**（`df05026`）：环绕远山、城市天际线（夜间万家灯火）、停车场车阵、飘动旗杆、塔台旋转雷达。
- **空中点缀**（`5321607`）：白天盘旋拍翼的飞鸟群、高空缓慢盘旋的直升机。

### 第 4 阶段 · 一处真实的插曲：穿模 bug（`fbdd88a`）

> 我的原话：**「飞机在停机坪会穿模。」**

这正是「AI 盲开发 + 人验收」协作闭环最生动的一幕：Claude 看不到画面，上一轮把地服车辆摆到了机翼正下方，1.6 米高的行李拖车从机翼里穿了出来。我一句话点出症状，Claude 据「机身中心 (gx,0,-10)、机头 +z、缩放 1.15」反推机翼/机身包络，把地服重排到**机尾后方 / 翼尖外侧 / 机头前方**的安全区，并让廊桥改为轻吻舱门而非插入。

> 💡 教训沉淀进了方法论：此后高风险、难自检的特性一律「默认关 + 可调 + 优雅回退」。

### 第 5 阶段 · 视觉细节三连（`3d326d8` → `e917945`）

> 我的原话：**「继续视觉细节：引擎热浪扰动、雨天地面真实镜面反射、航站楼内部夜间透光。」**

- **航站楼夜间透光**（`3d326d8`）：幕墙暖色室内辉光 + 楼体夜间自发光，入夜「亮起来」。
- **引擎/跑道热浪扰动**（`81bc61b`）：合成器里加屏幕带状折射后期，模拟热气升腾。
- **雨天地面镜面反射**（`e917945`）：天空专用 CubeCamera 渲染天空立方图，路面挂 envMap，雨天经抬高 `envMapIntensity` + 降糙显现，映出天穹/日月/云，干燥时零开销。

### 第 6 阶段 · 多端适配（`01f4811` → `c2d59a0`）

> 我的原话：**「你需要将项目适配手机端、平板端和电脑端……比如一些面板可回收展开等等。」**

- **触控 + 响应式 + 可收展面板**（`01f4811`）：发现并修复一个关键缺陷——此前是**纯鼠标事件，触屏根本无法操作 3D**。改为统一 Pointer 事件：单指拖拽旋转、双指捏合缩放、轻点选机；各面板可一键收展；≤1024 平板 / ≤760 手机断点重排。
- **手机抽屉菜单**（`0196046`）：顺着我「可以的」，把手机顶栏改为固定单行 + ☰ 下拉抽屉，顶栏高度从此可控。
- **收展动画 + 选完即收**（`c2d59a0`）：面板收展由硬切改为 `max-height` 平滑过渡，抽屉淡入下滑，点任一控件即自动收起；并尊重「减少动态效果」系统偏好。

### 第 7 阶段 · 写实冲刺：「我想要和真的一样」（`fd79061` → `f7af705`）

> 我的原话：**「继续回到视觉仿真继续优化，我想要和真的一样。」**

- **道面真实化**（`fd79061`）：把纯色块道面换成程序化沥青/混凝土纹理 + 法线微起伏 + 接地带轮胎胶痕（全 canvas 生成，零外部依赖）。

随后 Claude 提了 A/B/C/D 四个方向，我**全要**：

- **A 飞机涂装**（`7c0da3d`）：成排客舱舷窗（夜间发光）+ 川航红腰线 + 引擎进气整流锥。
- **B 运动真实化**（`d8d9c41`）：空中（仅离地后）转弯压坡度、平滑抬前轮/接地拉飘——纯渲染层实现，**不动内核**。
- **C SSAO 环境光遮蔽**（`ef29c98`）：深度通道 + 螺旋采样近似遮蔽。因盲调风险最高，**默认关闭**，交我开启评估。
- **D 太阳光柱**（`f7af705`）：屏幕径向光晕，仅太阳在画面且地平线之上时显现。

> 同时坦白：**真体积云**（光线步进）开销大、盲调极易翻车，未纳入；大气透视由现有 `FogExp2` 提供。

## 五、运行与复刻

```bash
# 方式一：双击 index.html（需联网加载 CDN 上的 three.js）
# 方式二：本地静态服务器
npx serve .
# 跑内核单元测试（零 npm 依赖）
npm test
```

**控制台实时调参速查**（F12）：

```js
AEROTWIN_FX.cfg.bloom.strength = 1.2;  AEROTWIN_FX.apply();   // 泛光
AEROTWIN_FX.cfg.dof.strength   = 0.4;  AEROTWIN_FX.apply();   // 移轴景深
AEROTWIN_FX.cfg.godray.strength= 0.5;  AEROTWIN_FX.apply();   // 太阳光柱
AEROTWIN_FX.cfg.ssao.enabled   = true; AEROTWIN_FX.apply();   // 开启 SSAO 评估
AEROTWIN_FX.cfg.enabled        = false;                       // 一键关全部后期
```

## 六、一句话总结

> 从一个**可复现、可测试、零依赖**的仿真内核出发，用「**方向菜单 → 全做 → 一次一提交 → 人眼验收 → 实时调参**」的人机结对节奏，在 28 次提交里把它打磨成了一座**会呼吸的微缩机场**——而真正让它没跑偏的，是那条始终没被打破的铁律：**视觉怎么炫，内核与那 9 个测试一直稳。**
